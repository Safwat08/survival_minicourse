# What is an Event?

An **event** is the outcome of interest whose time of occurrence is being modeled. Events must be clearly defined and unambiguous before analysis.

## Examples in Oncology

| Event | Definition |
|---|---|
| Overall survival (OS) | Death from any cause |
| Disease-free survival (DFS) | Recurrence or death |
| Progression-free survival (PFS) | Disease progression or death |
| Local recurrence | Tumor recurrence at primary site |

## Types of Events

The examples above fall into two broad families — **recurrence** and **death** — each with finer subtypes that define different endpoints.

### Recurrence

Reappearance of cancer after treatment. By location:

- **Local recurrence** — at the original tumor site
- **Regional recurrence** — in nearby lymph nodes
- **Distant recurrence / metastasis** — in other organs

Recurrence endpoints drive **disease-free survival (DFS)** and **recurrence-free survival (RFS)** analyses.

### Death

- **Overall survival (OS)** — time from treatment to death from *any* cause. The gold-standard endpoint in oncology trials: unambiguous, no cause-of-death judgment needed.
- **Cancer-specific survival (CSS)** — time to death attributed to *the cancer*. More targeted, but requires accurate cause-of-death records and is subject to misclassification.

### Choosing the Right Endpoint

| Endpoint | Pros | Cons |
|---|---|---|
| OS | Unambiguous | Requires long follow-up |
| DFS / RFS | Earlier signal | Recurrence detection depends on imaging schedule |
| PFS | Usable in trials | Progression criteria can vary |

## Requirements for a Well-Defined Event

- **Binary**: Either happened or didn't
- **Irreversible**: Once it occurs, the clock stops
- **Precisely timestamped**: Date of scan, pathology report, or death record

## Composite Endpoints

Some endpoints bundle several sub-events into **one** event. PFS is the canonical example:

> **PFS event = progression *OR* death, whichever comes first.**

This stays a single binary indicator — you do **not** create a second event or a second time:

| Field | Rule |
|---|---|
| `T` | time to the **earliest** of {progression, death}; if neither, the censoring time |
| `δ` | `1` if progression *or* death occurred; `0` if censored |

Because death is already inside PFS, modeling "PFS" and "death" as two separate events double-counts death. Only split them apart if you genuinely need to distinguish event *types* — that is the competing-risks design in [`1.4_multi_events_and_competing_risks.md`](1.4_multi_events_and_competing_risks.md), and it uses one time plus an event-*type* label, still never two times per patient.

For an end-to-end stratification pipeline built on a composite PFS endpoint, see [`08_recipes/8.1_pfs_stratification.md`](../08_recipes/8.1_pfs_stratification.md).

## Event vs. Censoring

A subject either experiences the event (δ=1) or is censored (δ=0) — never both.
