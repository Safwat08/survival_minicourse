What is an Event?

An event is the outcome of interest whose time of occurrence is being modeled. The endpoint must define the event, the time origin, and the rules used to assign an event date before analysis begins.

Examples in Oncology

Event Definition
Overall survival (OS) Death from any cause
Disease-free survival (DFS) Recurrence or death
Progression-free survival (PFS) Disease progression or death
Local recurrence Tumor recurrence at primary site

The same event definition can produce different times depending on the time origin: diagnosis, surgery, treatment start, or a landmark visit. State the origin explicitly and ensure every predictor was available by that time.

Types of Events

The examples above fall into two broad families — recurrence and death — each with finer subtypes that define different endpoints.

Recurrence

Reappearance of cancer after treatment. By location:

  • Local recurrence — at the original tumor site
  • Regional recurrence — in nearby lymph nodes
  • Distant recurrence / metastasis — in other organs

Recurrence endpoints drive disease-free survival (DFS) and recurrence-free survival (RFS) analyses.

Death

  • Overall survival (OS) — time from treatment to death from any cause. It is a hard, unambiguous endpoint that avoids cause-of-death adjudication, but often requires long follow-up and can dilute disease-specific effects.
  • Cancer-specific survival (CSS) — time to death attributed to the cancer. More targeted, but requires accurate cause-of-death records and is subject to misclassification.

Choosing the Right Endpoint

Endpoint Pros Cons
OS Unambiguous Requires long follow-up
DFS / RFS Earlier signal Recurrence detection depends on imaging schedule
PFS Usable in trials Progression criteria can vary

Requirements for a Well-Defined Endpoint

  • Operational definition: specify exactly what counts as an event and who adjudicates it
  • Time origin: use the same clinically meaningful starting point for every subject
  • Dating rule: define whether the date comes from a scan, pathology report, clinical assessment, or death record
  • Censoring rule: state when an event-free subject’s follow-up ends
  • Analysis type: say whether only the first event is analyzed, events can recur, or patients can move through multiple states

In an ordinary first-event analysis the stored indicator is binary and the clock stops at the first qualifying event. That is a property of the analysis record, not a claim that all clinical events are irreversible or exactly observed. Recurrent events, recovery, and interval-censored events require extensions beyond the simple (T, \delta) setup.

Composite Endpoints

Some endpoints bundle several sub-events into one event. PFS is the canonical example:

PFS event = progression OR death, whichever comes first.

This stays a single binary indicator — you do not create a second event or a second time:

Field Rule
T time to the earliest of {progression, death}; if neither, the censoring time
δ 1 if progression or death occurred; 0 if censored

Because death is already inside PFS, modeling “PFS” and “death” as two separate events double-counts death. Only split them apart if you genuinely need to distinguish event types — that is the competing-risks design in 1.4_multi_events_and_competing_risks.md, and it uses one time plus an event-type label, still never two times per patient.

For an end-to-end stratification pipeline built on a composite PFS endpoint, see 06_recipes/6.1_pfs_stratification.md.

Event vs. Censoring

For the recorded endpoint, a subject contributes either an observed event (\delta=1) or a censored observation (\delta=0). A patient may later experience the event after censoring, but that later time is unknown to the analysis.
